## Reliability improvement

Resumable uploads previously had no terminal condition when an append failed with a retryable error while the provider's cursor lookup repeatedly returned the same valid offset. A partial provider outage could therefore keep a large save pending forever.

The provider-independent upload adapter now:

- limits consecutive reconciliations that report no byte progress (four by default);
- applies exponential backoff between no-progress retries;
- resets the budget whenever either an append or provider reconciliation advances the cursor;
- rejects a regressed or invalid provider cursor instead of resending already acknowledged bytes;
- accepts an `AbortSignal` so callers can cancel during retry/reconciliation;
- reports structured recovery progress and a final offset/attempt count.

Dropbox inherits these protections without provider-specific project logic. Existing interrupted-upload cursor recovery and revision-controlled ambiguous-final-commit verification are unchanged.

## Compatibility, data-loss risk, and rollback

- No project schema, serialized format, cloud path, OAuth flow, or provider API changed.
- Healthy uploads and uploads whose remote cursor advances retain their existing behavior.
- During a sustained partial outage, a save now fails visibly after four same-offset recoveries instead of remaining pending indefinitely. The source project and previous cloud revision are not deleted or overwritten by this terminal failure.
- Rollback is a single commit revert; no migration or data repair is required.

## Exact human verification

Using only an approved disposable Dropbox account and synthetic project:

1. Connect, list, open, save, rename, disconnect, and reconnect a project; verify revision and feature count after reopening.
2. Save a synthetic project larger than 24 MiB and interrupt a middle upload chunk after the provider accepts zero bytes.
3. Confirm four retry/reconciliation cycles occur with increasing delays, then an explicit recoverable failure appears rather than a permanent spinner.
4. Retry after restoring the network and reopen the project; verify feature count, geometry, attributes, and the previous revision remain intact.
5. Repeat with partial byte acceptance and confirm upload resumes at the provider offset without duplicate/corrupt bytes.
6. Exercise a stale revision conflict plus auth, quota, offline, and final-response-loss cases; confirm each produces the existing recovery guidance and no data loss.
